# TICKET-01: Orphan data trong bảng prescriptions

**Phát hiện:** E2E test ngày 2026-07-21 trên `https://his.atds.com.vn/`

## Severity & Priority
- **Severity:** Medium
- **Priority:** P2 (xử lý trong sprint hiện tại)

## Mô tả
Response API `GET /api/v1/prescriptions?page=1` trả về các record có:
- `encounter_id = 00000000-0000-0000-0000-000000000000`
- `patient_id = 00000000-0000-0000-0000-000000000000`

Đây là dấu hiệu **orphan data** (đơn thuốc không thuộc lượt khám / bệnh nhân nào) hoặc **seed dữ liệu bị lỗi**.

## User Story
**Given** tôi là quản trị viên dữ liệu,
**When** tôi query bảng `prescriptions`,
**Then** mọi đơn thuốc phải tham chiếu tới `encounter_id` và `patient_id` hợp lệ (tồn tại trong bảng cha).

## Acceptance Criteria
- [ ] Điều tra nguồn gốc record GUID `0000...`: do seed hay do bug service khi insert.
- [ ] Nếu là seed sai → xóa/sửa seed, viết migration cleanup idempotent trong `db/migrations/`.
- [ ] Nếu là orphan thật → bổ sung **FK constraint** `prescriptions.encounter_id → encounters.id`, `prescriptions.patient_id → patients.id` (kèm rule `ON DELETE RESTRICT`).
- [ ] Thêm validation ở service layer: reject insert nếu `encounter_id`/`patient_id` = GUID rỗng hoặc không tồn tại.
- [ ] Viết unit test đảm bảo không thể tạo prescription orphan.
- [ ] Chạy script kiểm tra toàn bảng `prescriptions` sau fix, log số record đã xử lý.

## Edge case
- Tenant khác nhau: cleanup phải scope theo `tenant_id`.
- Soft-delete: nếu encounter cha bị soft-delete, prescription giữ trạng thái nào?

## Assignee
- Backend (chính) + Data-agent (hỗ trợ query & migration)

## Effort
**M** (0.5–1 ngày dev + 0.5 ngày test)
